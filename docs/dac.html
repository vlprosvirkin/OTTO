Architecture Spec

Decentralized Autonomous Corporation

How autonomous treasury management works — from fund creation to invoicing, payroll, budgeting, yield optimization on idle balances, and emergency controls

Participants

💰

Investor / LP Holder

Deposits funds into the treasury, receives LP tokens representing their share, and governs the corporation through token-weighted voting. A single role — every contributor is also a decision-maker.

📊

Manager

Operates the treasury — issues invoices, approves expenses, runs payroll, and manages budgets. Cannot withdraw funds directly.

🛡

Guardian

Emergency oversight. Can freeze the entire corporation in case of an emergency. Controls global safety parameters.

Corporation Lifecycle

1

Corporation Creation (Factory)

The Factory deploys a complete set of contracts for each DAC:

  • Pool (Treasury) — holds all corporation assets
  • LP Token — non-transferable share of participation
  • Voting — governance and decision-making module
  • Configuration — corporation rules (fees, limits, windows, budgets)

After creation, the Factory renounces ownership — the corporation becomes fully self-governing.

2

Fundraising

Investor → deposits funds → Pool values contribution via oracle → → mints LP tokens proportional to deposit → minus entry fee to Manager

Constraints:

  • Only during the designated fundraising window
  • Minimum and maximum deposit size enforced
  • Private corporations — whitelist-only addresses
  • LP tokens are frozen for N hours after deposit
3

Treasury Operations (Manager)

Manager → submits an expense → Pool validates: ✓ Recipient is on the approved payee whitelist ✓ Expense category is within the approved budget ✓ Amount is within per-tx and daily spending caps ✓ Invoice reference is attached and valid → Pool executes the payment on behalf of the corporation

The Manager cannot withdraw assets freely — every outflow must pass through approved channels: invoices, payroll schedules, or budgeted expense categories. Validators decode each transaction and verify its legitimacy.

Operation types:

  • Invoices — pay external vendors and service providers against submitted invoices
  • Payroll — recurring salary payments to employees on a set schedule
  • Budgeting — allocate funds to departments or categories with spending caps
  • Expense approvals — one-off payments within pre-approved budget lines
4

Yield Optimization (Idle Balance Management)

Pool monitors idle USDC balance → ✓ Calculates reserve needed for upcoming payroll + invoices ✓ Sweeps excess USDC into USYC (tokenized US T-bills) ✓ USYC accrues yield continuously (~5% APY) ✓ Auto-redeems USYC back to USDC when liquidity is needed

Idle treasury balances are never left sitting unproductive. The Pool automatically invests surplus USDC into USYC (Hashnote tokenized T-bills), earning risk-free yield on idle funds while maintaining enough liquid USDC to cover short-term obligations.

How it works:

  • Reserve threshold — Pool keeps enough liquid USDC to cover N days of projected expenses
  • Auto-sweep — any balance above the threshold is deposited into USYC
  • Auto-redeem — when liquid balance drops below threshold, USYC is redeemed back to USDC
  • Yield accrual — USYC appreciates in value continuously, increasing the total treasury value
Example: $100k treasury, $20k monthly burn → Reserve: $30k USDC (liquid, 1.5x monthly burn) → Invested: $70k in USYC earning ~5% APY → Yield: ~$3,500/year on idle funds
5

Withdrawal

Investor → requests withdrawal → Pool calculates: 1. Proportional share of all assets 2. Performance fee (if LP price increased since entry) 3. Rage quit fee (if outside withdrawal window) → Burns LP tokens → Transfers asset share to investor

6

Manager Fees

Fee When Logic
Entry fee On deposit % of minted LP tokens
Management fee Continuous Annual % of AUM, accrued every second
Performance fee On investor withdrawal % of profit (if LP price increased)

The Manager claims fees via a separate transaction — receiving a proportional share of all pool assets.

7

Governance (Voting)

LP Holder → creates proposal → Token-weighted vote → → If quorum reached and majority in favor → Pool executes action

What can be proposed:

  • Replace the Manager (if allowed by configuration)
  • Remove a payee from the approved whitelist
  • Adjust budget caps or spending limits
  • Add or remove approved expense categories
  • Change yield reserve threshold or disable auto-sweep

Constraints:

  • The Manager can submit any type of proposal
  • Regular investors — only the actions listed above
  • A minimum LP token share is required to create a proposal
  • Votes are counted by snapshot (at the moment of proposal creation)
8

Emergency Stop

Guardian → activates emergency stop → ✗ All operations blocked (invoices, payroll, expenses) ✗ Deposits blocked ✗ New proposals blocked ✓ Withdrawals WITHOUT penalty — allowed

Core Principle

The Manager operates, but cannot extract funds. Idle treasury balances are never left unproductive — surplus USDC is automatically swept into USYC (tokenized T-bills) to earn yield. Investors cannot run day-to-day operations, but can vote to replace the Manager or restrict their authority. The Guardian can freeze everything in an emergency.

Chat-Based Governance with OTTO

On-chain governance is powerful but creates friction: users need wallets, gas, and must navigate dApp UIs to vote. OTTO eliminates this friction by acting as a governance relay inside a Telegram group chat where all corporation participants communicate.

How It Works

OTTO is added to the corporation's Telegram group chat as a bot. All participants — investors, manager, guardian — interact in the same chat. OTTO recognizes each user by their Telegram ID, maps them to on-chain wallets, and tracks their roles and voting weight in real time.

1

Wallet Linking

Each participant sends /link 0x... in the group or DM. OTTO verifies ownership via signature challenge and binds Telegram ID ↔ wallet address.

2

Role & Share Sync

OTTO reads the LP Token contract to determine each linked user's voting weight and role (investor, manager, guardian). Updated automatically on every deposit/withdrawal.

3

Propose via Chat

Any eligible participant types a proposal in plain language. OTTO parses it, validates eligibility, creates an on-chain proposal, and posts a vote card in the group.

4

Vote via Reply

Participants vote by replying to OTTO's vote card with a single message. One message = one vote, weighted by LP token share. No wallet popup, no gas.

Participant writes proposal in Telegram → OTTO validates eligibility (role + min LP share) → Creates on-chain proposal via Voting contract → Posts vote card in group chat → Participants reply to vote (FOR / AGAINST) → OTTO records votes with LP-token weighting → Quorum reached? → OTTO executes on-chain → Posts result summary to the group

Delegation model: participants delegate voting authority to OTTO's operator wallet. OTTO submits votes on-chain on behalf of each user, using the delegated authority. This avoids the need for each participant to sign individual transactions.

What OTTO tracks per user:

  • Telegram ID — unique identifier in the group chat
  • Wallet address — linked and verified via signature
  • Role — investor, manager, or guardian (read from contracts)
  • LP token balance — determines voting weight
  • Vote history — all past votes for transparency and audit

What participants can do in chat:

  • Investors — propose manager replacement, budget changes, whitelist edits; vote on all proposals
  • Manager — propose any action; cannot vote on their own replacement
  • Guardian — trigger emergency stop directly via command
  • Anyone — ask OTTO for treasury status, balances, active proposals, vote results

Implementation Specification

1. Bot Setup & Group Integration

  • OTTO runs as a Telegram Bot (via BotFather) added to the corporation's group chat
  • Bot requires admin permissions to read all messages and manage pinned posts
  • Each DAC pool address is bound to exactly one Telegram group (poolAddress ↔ chatId)
  • OTTO listens to all messages but only processes commands (/link, /propose, /status, /emergency) and vote replies
  • Regular conversation between participants is unaffected — OTTO stays silent unless addressed

2. Wallet Linking & Verification

User: /link 0xAlice...7890 OTTO: Please sign this message to verify ownership: "Link Telegram 483291 to DAC pool 0xPool...1234" User: <sends signature via DM or web link> OTTO: ✓ Wallet linked! You are Investor with 14.2% voting power.
  • OTTO generates a unique nonce-based message for the user to sign with their wallet
  • Signature can be submitted via a lightweight web page (deeplink from Telegram) or pasted as text
  • OTTO verifies ecrecover(message, signature) == claimed address
  • One wallet per Telegram ID; re-linking replaces the previous binding
  • Linking can also happen via DM to OTTO (private) rather than in the group

3. Role & Balance Sync

  • On link, OTTO reads lpToken.balanceOf(address) and lpToken.totalSupply() to compute voting weight
  • OTTO subscribes to Transfer events on the LP Token contract to update balances in real-time
  • Role detection: Manager = pool.manager(), Guardian = pool.guardian(), everyone else = Investor
  • If an investor's balance drops to zero (full withdrawal), their voting right is revoked until they re-deposit
  • OTTO periodically reconciles its local state with on-chain data (every 5 minutes as a safety net)

4. Data Model (per user)

{ telegramId: number, // Telegram user ID chatId: number, // Corporation group chat ID walletAddress: string, // 0x... linked wallet verified: boolean, // signature verified? role: "investor" | "manager" | "guardian", lpBalance: bigint, // LP token balance (wei) votingPower: number, // % of total supply linkedAt: timestamp, // when wallet was linked votes: Vote[] // history of all votes cast }

5. Proposal Lifecycle

Phase Duration What Happens
Creation Instant OTTO validates eligibility, creates on-chain proposal, pins vote card in chat
Voting 24–72h (configurable) Participants reply FOR/AGAINST; OTTO tallies in real-time, posts progress updates
Finalization Instant Quorum met → OTTO executes on-chain; not met → proposal expires
  • Only one active proposal at a time per category (prevents spam)
  • Duplicate votes are ignored — first vote counts
  • OTTO posts a running tally after each vote: FOR: 42% / AGAINST: 12% / Quorum: 51%
  • If quorum is reached before deadline, OTTO can execute immediately (configurable)

6. On-Chain Execution (Delegation Model)

  • During wallet linking, users sign a one-time delegation tx: votingContract.delegate(ottoWallet)
  • OTTO accumulates votes off-chain, then submits one batched transaction: votingContract.submitVotes(proposalId, votes[])
  • The Voting contract verifies that OTTO holds valid delegation from each voter
  • After votes are recorded on-chain, if quorum is met, OTTO calls votingContract.execute(proposalId)
  • All on-chain activity is reported back to the group with transaction hashes

7. OTTO Commands Reference

Command Who Action
/link 0x... Anyone Link Telegram ID to wallet address
/status Anyone Show treasury balance, active proposals, yield status
/propose <action> Investor / Manager Create a governance proposal in natural language
/tally Anyone Show current vote count for active proposal
/my Linked user Show your wallet, role, LP balance, voting power
/members Anyone List all linked members with roles and voting shares
/emergency Guardian only Trigger emergency stop on the pool contract

8. Security Considerations

  • Wallet linking requires cryptographic signature — no one can impersonate another wallet
  • OTTO's delegation can be revoked at any time by calling delegate(address(0)) on-chain
  • Vote snapshots are taken at proposal creation time — buying LP tokens after a proposal is live does not grant voting power on it
  • Guardian's /emergency bypasses governance — it is a direct on-chain call, not a proposal
  • All vote records are stored on-chain and auditable; Telegram chat serves as a human-readable mirror
